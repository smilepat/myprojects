# Part 1. 수능 영어가 진짜로 요구하는 역량 (12분)

## 슬라이드 3 — 통념 vs. 실제

**통념:** "수능 영어 = 교과과정 충실한 학습(?): 단어 + 문법 + 독해 + 문제풀이"
**실제:** 수능 영어 = **대학 1학년 수준 학술 독해를 시간 압박 하에 수행**하는 능력

| 지표 | 수능 영어 | 비고 |
|------|----------|------|
| 텍스트 난이도 | Lexile 1100–1300L | 미국 대학 입학 수준 |
| 시간 제약 | 70분 / 45문항 | 문항당 약 93초 |
| 인지 수준 | 추론·요약·함의 추론 | Bloom's Taxonomy 상위 |
| 어휘 범위 | AWL(학술어) + 추상어 다수 | 평가원 출제 패턴 10년 누적 |

> **노트:** "청중들에게 강조 — 이건 '영어 시험'이 아니라 *시간 제약 하의 인지 시험*입니다. 단순 문제풀이 앱이 시장에서 통하지 않은 이유, 그리고 우리가 다른 카테고리에서 경쟁하는 이유가 여기 있습니다."

---

## 슬라이드 4 — 우리의 역량 분해 모델: 5D Diagnostic

**ET-Craft 5차원 진단 프레임 (D1–D5)**

| 차원 | 이름 | 측정 대상 | 예시 |
|------|------|-----------|------|
| **D1** | Form | 어휘 형태·스펠링 인식 | "achieve / achievement 굴절 인식" |
| **D2** | Meaning | 의미 매핑 | 한국어 ↔ 영어 의미 즉시 인출 |
| **D3** | Context | 문맥 활용 | 문장 내 적절 어휘 선택 |
| **D4** | Collocation | 연어·관용 결합 | "make a decision" 자연성 판단 |
| **D5** | Usage | 담화·전략 사용 | 추론·시간 분배·메타인지 |

> **노트:** "기존 학습지·인강은 '어휘 외우기'를 한 덩어리로 다룹니다. 우리는 **5개 차원으로 분해**합니다. 'D1에서 막힌 학생'과 'D5에서 막힌 학생'은 처방이 완전히 다릅니다. 분해 자체가 경쟁 우위입니다."

---

## 슬라이드 5 — 결정적 변수: D5 (담화·전략)

**현장 관찰:**
- D1–D4가 충분해도 **시간 압박 하에선 정답률 급락**
- 상위권–최상위권의 결정적 차이는 D5 (메타·전략 영역)
- 교과과정·교과서 어디에도 **D5는 명시적으로 다루지 않음**

**비즈니스 시사점:**
- D5는 **개인별 인지 패턴 데이터**가 없으면 못 가르침
- 인간 1:1 코칭 영역인데 **단가가 안 맞음**
- → **여기가 AI의 자리, 우리의 카테고리 정의 지점**

> **노트:** "오늘 강연의 **첫 번째 비즈니스 포인트**입니다. D5 영역은 시장에 기존 솔루션이 없습니다. 기존 사교육이 못 하는 게 아니라 *경제성 있게 못 하는* 겁니다. 우리는 풀이 로그 데이터로 이걸 합니다."

## 슬라이드 7 — 교과과정은 무엇을 약속하는가

**2022 개정 교육과정 영어과 성취기준 예시**

> `[12영어I-04-02]` 다양한 주제에 관한 글을 읽고 함축적 의미를 추론한다.

**구조적 한계:**
- 일반적 can-do 진술 — **"추론한다"의 합격선·평가 루브릭 부재** (정답률 60%인지 90%인지, 채점자가 무엇을 보고 "추론 성공"이라 판단하는지 어디에도 명시되지 않음)
- 텍스트 난이도 명시 없음 (Lexile·CEFR 등 정량 지표 미사용)
- 시간 제약 명시 없음 (1문항당 몇 초 안에 처리해야 하는지 무규정)
- **"무엇을"** 은 있지만 **"어느 수준에서, 얼마나 빠르게, 얼마나 정확하게"** 는 없음

> **노트:** "성취기준이 잘못됐다는 게 아닙니다. *측정 가능성*이 비어 있다는 거죠. EdTech가 부가가치를 만드는 자리가 바로 거기입니다."

## 슬라이드 8 — 학년간 점프: Lexile 데이터

**Lexile 기준 학년별 난이도**

| 단계 | 권장 텍스트 난이도 | 비고 |
|------|--------------------|------|
| 중3 | 600–800L | 교과서 평균 |
| 고1 | 800–900L | 교과서 |
| 고2 | 900–1000L | 모의고사 |
| **수능** | **1100–1300L** | 학술 산문 |

> **노트:** "이게 갭의 정량적 실체입니다. 정규 수업 시간으로 메우는 건 *수학적으로* 불가능합니다. 그래서 사교육 의존도가 구조적으로 형성되고, 동시에 우리가 진입할 시장 자체가 형성됩니다."

## 슬라이드 9 — 평가 방식의 비대칭

| 항목 | 학교 내신 | 수능 |
|------|----------|------|
| 주요 문항 유형 | 어휘·문법 객관식, 본문 암기형 | 추론·빈칸·함의·요약 |
| 측정 인지 수준 | Bloom 하위 (기억·이해) | Bloom 상위 (분석·평가) |
| 텍스트 출처 | 교과서 | 학술지·논픽션·외부 지문 |
| 시간 압박 | 낮음 | 매우 높음 |
| D5 측정 | 거의 없음 | 사실상 결정 변수 |

**핵심:** "학교 1등급 → 수능 3등급" 현상의 구조적 이유.

> **노트:** "투자자 관점 — 학교 시스템이 수능 대비를 *못 하는* 게 아니라 *측정 자체를 안 하고 있다*는 겁니다. **측정이 비어 있는 곳에 시장이 있습니다.**"

## 슬라이드 10 — 갭의 3가지 정량 지표 (한 장 요약)

1. **난이도 갭:** Lexile 200–400L 점프 (고2 → 수능)
2. **시간 갭:** 평소 학습 → 문항당 93초 압축
3. **인지 갭:** Bloom 하위 → 상위

**시장 함의:**
- 3가지 갭은 **개인별로 비대칭** → 일괄 커리큘럼으로 못 메움
- 개인화·적응형 학습의 정당성은 **취향이 아니라 수학적 필연**

> **노트:** "여기까지가 진단입니다. Part 3로 갑니다."

## 슬라이드 11 — 해법의 4가지 설계 원칙

1. **분해 (Decompose):** 역량을 5D × 3단계 micro-skill로 분해
2. **진단 (Diagnose):** IRT Rasch 모델 기반 적응형 진단
3. **생성 (Generate):** 44개 수능 유형별 2-Stage LLM 파이프라인
4. **순환 (Iterate):** 학습-평가-재진단 루프 + 프롬프트 A/B 운영

> **노트:** "여기서 중요한 건 우리가 *콘텐츠를 모으는* 회사가 아니라 *콘텐츠를 생성·검증하는* 회사라는 점입니다. 콘텐츠 비용 곡선이 다릅니다."

## 슬라이드 14 — IRT 적응형 진단: 1PL Rasch부터 시작

**왜 현재는 1PL Rasch인가:**
- 런칭 전엔 사용자 정답률 로그(Empirical Data) 없음 → **Cold-Start**
- 가장 학술적으로 검증된 대안: **빈도 순위(Zipf's law) → Z-score 변환**
- 9,017개 문항 전부에 `a = 1.0` 고정, `b = −3.00 ~ +3.67` 분포

**런칭 후 자가 진화 로드맵:**

| Phase | 전환 | 트리거 |
|-------|------|--------|
| **현재 (1PL)** | 빈도순위 기반 b_value | — |
| Phase 1 | 실제 정답률 기반 b_value 재추정 (MLE) | 수만 건 정답 로그 누적 |
| Phase 2 | **2PL 확장** — a_value 해금, 매력적 오답 문항 자동 발굴 | 충분한 변별도 데이터 |
| Phase 3 | **Warm-Start** — 학년·CEFR 정보로 초기 θ 추정 | 사용자 프로필 풀 확보 |

> **노트:** "정직하게 말씀드립니다 — 현재는 1PL입니다. 하지만 이건 **약점이 아니라 설계**입니다. 듀오링고도 같은 경로를 거쳤습니다. 사용자 로그가 쌓이는 매 분기마다 모델이 자가 진화하도록 파이프라인이 이미 구축돼 있다는 게 핵심입니다."

## 슬라이드 15 — 44개 유형 × 2-Stage 생성 파이프라인

**핵심 성과:** 문항 1개 생성 **20초 이내**, 교사 제작 시간 **85% 절감**

```
[교사 입력: 지문 + 난이도]
        ↓
[Type Router] — 규칙 45% + LLM 55% 가중합 → 적합 유형 5–10개 추천
        ↓
[Stage 1: 지문 분석] — 빈칸 후보·담화 구조·핵심 어휘 추출
        ↓
[Stage 2: 문항 생성] — 유형별 전용 프롬프트 + Pydantic 검증
        ↓
[Postprocess] — HTML sanitize, 스마트쿼트 정규화
        ↓
[Streaming Response, 8초 하트비트] + 크레딧 차감
```

**운영 인프라:**
- 프롬프트 3-환경 분리: `production/` · `staging/` · `archive/`
- `/api/generate/compare` — production vs staging 블라인드 A/B 비교
- 약 3일 주기의 staging → production 롤아웃 사이클

> **노트:** "여기서 두 번째 비즈니스 포인트 — 우리가 콘텐츠를 *생성*한다는 건 단순히 '많이 만든다'가 아닙니다. **A/B 실험 인프라가 있기 때문에 시간이 지날수록 품질이 복리로 좋아진다**는 뜻입니다. 인강은 강사 의존, 학습지는 출판 의존, 우리는 데이터 의존입니다."

## 슬라이드 17 — 🆕 LMS 어댑터: B2G 도입 장벽 우회

**한국 학교 시장의 진짜 장벽:**
- 모든 학교에 이미 Java Spring 기반 학사 시스템이 있음
- 신규 시스템 도입 = IT 부서 검토 6–12개월
- "기존 시스템 교체"는 정치적으로 불가능

**ET-Craft 해법: 사이드카 어댑터**

```
┌──────────────┐   HTTPS   ┌───────────────┐   REST   ┌─────────────┐
│ Next.js 15   │──────────▶│  FastAPI      │─────────▶│  Java API   │
│ (교사 UI)    │           │  (AI 엔진)    │          │ (기존 LMS)  │
└──────┬───────┘           └───┬───────┬───┘          └─────────────┘
       │                       │       │
  Docker                   Redis   Azure OpenAI / Gemini
```

**효과:** 학교 IT 부서가 *기존 시스템을 한 줄도 안 건드림*. 도입 검토 기간 6개월 → 4주.

> **노트:** "GTM 메시지로 정리하면 — *'기존 LMS는 그대로 두세요. 저희가 위에 얹습니다.'* 학교 의사결정자가 가장 듣고 싶어 하는 문장입니다. 우리는 이 한 문장을 위해 아키텍처를 설계했습니다."

## 부록 A — Q&A 대비

**Q1. "왜 GPT-4-turbo인가? 한국 모델은 안 보는가?"**
- 현재는 **엔터프라이즈 데이터 처리 계약 + SLA**가 가능한 Azure OpenAI를 주력.
- B2G 도입 시 학교 IT 부서의 1순위 검토 항목이 *데이터 주권*이며, Azure가 한국 리전·계약 모두 충족.
- 한국 모델(EXAONE, Solar Pro 등)은 **2027년 자체 데이터 기반 sLLM 파인튜닝 시점에 재평가** 예정.

**Q2. "왜 1PL Rasch부터? 2PL/3PL이 더 정밀하지 않나?"**
- Cold-Start 단계에서 2PL은 *과적합 위험*이 큼.
- 1PL은 **언어학적 Zipf's law를 Z-score로 치환한 검증된 기준선**.
- 사용자 정답 로그가 누적되면 자동으로 2PL로 확장하는 파이프라인이 이미 설계되어 있음.

**Q3. "저작권 이슈는?"**
- 수능 기출: 한국교육과정평가원 저작물 — 학습 목적 인용 범위 준수.
- ET-Craft 생성 콘텐츠: 6-규칙 엔진 사전 판별 + Grade A/B만 자동 승인.
- 외부 지문 처리: light/medium/deep 변환 제안으로 교사 자기검열 보조.

**Q4. "교사 대체 논란은?"**
- ET-Craft는 *대체*가 아니라 *증강* 포지셔닝.
- 핵심 기능(2차 검증 게이트)이 *교사 승인 없이 학생 배포 불가* 구조.
- 크레딧 단가가 인간 강사 단가보다 100배 낮아 *대체*가 아니라 *교사 시간 회수*임이 명백.

**Q5. "Azure OpenAI 의존도가 너무 높지 않나?"**
- LLM 추상화 레이어(`llm_client.py`) + Gemini 폴백 이미 구현.
- 멀티 벤더 전략으로 단일 벤더 리스크 분산.
- 장기 sLLM 파인튜닝으로 의존도 자체 감소 로드맵.
